# 因子库管理页面设计方案

> **文档版本** 0.1 · 2026-03-17 · 状态：*RFC*

---

## 一、页面概览

![因子库管理页面 Mockup](/Users/liukun/.gemini/antigravity/brain/5363dba4-befb-48b1-bfaf-1fb69f775ead/factor_library_page_1773714827397.png)

因子库管理页面是一个**独立的后台页面**（路由 `/admin/factors`），与现有 Phase 5 工作台平级。核心功能：

| 功能区域 | 职责 |
|---|---|
| **因子库浏览** | 按维度/状态/标签浏览和搜索已入库因子 |
| **待审核门禁** | 显示所有待审核因子，人工决定是否入库 |
| **因子提取** | 贴入文章链接，触发 AI 提取因子，预览后入库 |
| **因子编辑** | 查看/编辑因子详情，管理生命周期 |

## 三、核心组件设计

### 3.1 因子卡片（FactorCard）

因子卡片是页面的核心组件，有三种变体：

#### 变体 A：库中因子卡片

```

│  反直觉数据钩子                    [编辑] │

│  └────┘ └────┘                          │

│  #数据驱动  #悬念                        │
└─────────────────────────────────────────┘
```

**规格**：
- 卡片宽度：自适应网格 `minmax(280px, 1fr)`
- 内边距：16px
- 圆角：8px
- 背景：`#252540`（暗色主题） / `#fff`（亮色主题）
- Hover：上移 2px + 加深阴影
- 维度 badge：彩色 pill，不同维度不同颜色
- 状态 badge：有效=绿，草稿=灰，废弃=红

## 四、交互规范

### 4.1 状态反馈

| 操作 | 反馈方式 |
|---|---|
| 入库因子 | 卡片绿色闪烁 → 从待审核列表滑出 → 左栏因子库刷新 |
| 驳回因子 | 卡片红色闪烁 → 从列表滑出 → toast 提示"已驳回" |
| 提取因子 | 按钮变为 loading → 下方渐入进度指示器 → 结果列表渐入 |
| 搜索/筛选 | 300ms debounce → 卡片网格 fade 切换 |
| 编辑保存 | modal 关闭 → 对应卡片内容就地更新（无需刷新） |

### 4.2 空状态设计

| 场景 | 展示内容 |
|---|---|
| 因子库为空 | 居中图标 + "因子库还是空的" + "手动创建" / "从文章提取" 两个按钮 |
| 待审核为空 | "🎉 没有待审核的因子" + 次要文字 "新提取的因子会出现在这里" |
| 搜索无结果 | "没有找到匹配的因子" + "试试其他关键词" |
| 提取无因子 | "😅 未能从这篇文章中提取到有价值的因子" + "可能文章风格较为常规" |

### 4.3 批量操作

待审核列表支持批量操作：
- 顶部 "全选" checkbox
- 选中后底部出现浮动操作栏："批量入库 (N)" / "批量驳回 (N)"

### 4.4 键盘导航

| 快捷键 | 功能 |
|---|---|
| `/` | 聚焦搜索框 |
| `Esc` | 关闭 modal / 取消选择 |
| `Enter` | 待审核卡片聚焦时 = 入库 |
| `Delete` | 待审核卡片聚焦时 = 驳回 |
| `↑` / `↓` | 在待审核列表中导航 |

## 六、技术实现方案

### 6.1 路由与入口

```python
# app/api/admin.py 新增路由
@router.get("/admin/factors", response_class=HTMLResponse)
async def factor_library_page(request: Request):
    return HTMLResponse(content=_render_factor_page())
```

### 6.2 前端架构

与现有 `admin.py` Phase 5 工作台保持一致的技术栈：
- **HTML**：内联在 Python HTMLResponse 中
- **CSS**：`<style>` 标签，复用现有 Design Token
- **JS**：`<script>` 标签，Vanilla JS + fetch API

### 6.3 依赖的后端 API

| API | 用途 |
|---|---|
| `GET /api/v1/factors` | 因子列表（筛选/搜索/分页） |
| `POST /api/v1/factors` | 手动创建因子 |
| `PUT /api/v1/factors/{id}` | 编辑因子 |
| `PATCH /api/v1/factors/{id}/status` | 状态变更（入库/驳回/废弃） |
| `POST /api/v1/factors/extract` | 提交文章链接触发提取 |
| `GET /api/v1/factors/extractions/{id}` | 获取提取结果 |
| `POST /api/v1/factors/extractions/{id}/confirm` | 批量确认提取结果入库 |

## 七、与方案文档的关系

本页面设计对应 [article-factor-library.md](./article-factor-library.md) 中的以下模块：

| 方案章节 | 本页面覆盖的功能 |
|---|---|
| 四、因子提取引擎 | 提取面板（底栏） |
| 八、数据模型 | 因子 CRUD + 状态管理 |
| 九、API 设计 | 所有因子相关 API |
| 十、Phase A | 因子库浏览 + 手动创建 + 审核门禁 |
| 十、Phase B | 提取面板 + 去重标记 |

**新增的人工审核门禁流程**：

```
AI 提取 / 自动抓取 → 待审核队列 → 人工审核（入库/编辑/驳回）→ 因子库
                                    ↑
                        人工手动创建 → (可选跳过审核，直接 active)
```

所有自动来源的因子**必须经过人工审核**才能进入因子库（status 从 `pending` → `draft` → `active`）。
人工手动创建的因子可以直接以 `draft` 状态入库，由创建者自行决定何时激活为 `active`。
