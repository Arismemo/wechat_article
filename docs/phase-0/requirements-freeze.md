# 阶段 0 需求冻结文档

更新时间：2026-03-07
状态：Active

## 1. 项目目标

本项目要做的是一个`微信公众号选题重构与草稿生产系统`，目标是把“复制文章链接 -> 服务器分析 -> 生成新稿 -> 存入公众号草稿箱”的链路做成稳定、可回溯、可人工审核的半自动流水线。

首版目标不是自动发布，也不是逐句改写，而是：

- 稳定接收微信文章链接
- 多源分析同题内容
- 重构出新的公众号文章
- 自动执行风控审稿
- 合格稿件进入草稿箱，等待人工发布

## 2. 核心用户流程

1. 用户在微信里看到一篇值得参考的文章。
2. 用户复制文章链接。
3. 用户双击 iPhone 背面，快捷指令读取剪贴板并调用服务端接口。
4. 服务端创建任务，异步抓取原文并做分析。
5. 系统检索同主题 3 到 5 篇参考内容，形成差异矩阵。
6. 系统先生成新选题 Brief，再生成标题、摘要、正文和封面短文案。
7. 系统执行相似度、事实、风险、风格和格式检查。
8. 合格稿件推送到公众号草稿箱。
9. 用户在后台浏览草稿、复核并手动发布。

## 3. 功能范围

### 3.1 必做功能

- `FR-001`：支持通过 iPhone 快捷指令提交文章链接。
- `FR-002`：支持服务端 URL 标准化、鉴权、幂等去重和任务入队。
- `FR-003`：支持抓取原始文章正文、标题、作者、发布时间、摘要和配图。
- `FR-004`：支持保存原文快照与清洗后的正文。
- `FR-005`：支持根据原文生成主题分析结果。
- `FR-006`：支持生成标准、热点、反向三类搜索 query。
- `FR-007`：支持搜索、筛选并抓取 3 到 5 篇同题参考内容。
- `FR-008`：支持形成差异矩阵和新选题 Brief。
- `FR-009`：支持生成标题组、摘要、大纲、正文和封面短文案。
- `FR-010`：支持执行自动审稿和综合评分。
- `FR-011`：支持将合格稿件转为 HTML 并写入公众号草稿箱。
- `FR-012`：支持查看任务状态、失败原因和草稿结果。
- `FR-013`：支持重跑、重生成和人工介入。
- `FR-014`：支持保存完整审计轨迹。

### 3.2 首版不做

- `NFR-OOS-001`：不做自动正式发布。
- `NFR-OOS-002`：不做句子级洗稿或同义替换。
- `NFR-OOS-003`：不做复杂封面自动设计。
- `NFR-OOS-004`：不做绕过访问限制的抓取。
- `NFR-OOS-005`：不做多账号、多团队协同。

## 4. 非功能要求

- `NFR-001`：任务必须可追踪，所有阶段状态清晰可见。
- `NFR-002`：关键失败必须可重试，可转人工处理。
- `NFR-003`：所有来源、生成、审稿和发布动作必须留审计日志。
- `NFR-004`：生成稿必须显著区别于来源文章结构，不能靠简单改写。
- `NFR-005`：高风险内容必须进入人工终审。
- `NFR-006`：系统默认只自动入草稿箱。
- `NFR-007`：密钥、令牌、Webhook 等敏感配置必须走环境变量或密钥管理，不写死在代码里。

## 5. 产品边界

### 5.1 输入边界

- 主输入是微信公众号文章 URL。
- 备用输入可以是分享菜单传入的 URL。
- 如果 URL 无法抓取，可进入人工补料分支。

### 5.2 输出边界

- 主要输出是公众号草稿箱文章。
- 中间输出包括：原文快照、同题素材、分析结果、差异矩阵、Brief、生成稿、审稿报告。

### 5.3 合规边界

- 默认不自动发布。
- 不输出未核实的高风险事实。
- AI 参与生成和审稿必须有留痕。

## 6. 成功标准

- 链接接收成功率 >= 95%
- 原文抓取成功率 >= 85%
- 草稿箱推送成功率 >= 95%
- 自动审稿后可进入草稿箱比例 >= 60%
- 每条任务可回溯到原链接、分析、Prompt、生成版本和草稿结果

## 7. 已确认决策

- 产品定义固定为`半自动选题重构与草稿生产系统`。
- 手机端主入口固定为`复制链接 + 双击背面 + 读取剪贴板`。
- 微信公众号链路固定为`自动入草稿箱 + 人工发布`。
- 内容生成策略固定为`同题重构`，不是`逐句改写`。
- 所有项目信息都必须以文档形式留存于 `docs/`。
