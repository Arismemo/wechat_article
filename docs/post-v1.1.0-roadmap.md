## 1. 规划前提

- 当前系统已经具备 Phase 4 写稿与审稿、Phase 5 审核台、Phase 6 反馈台、Phase 7 统一控制台与设置页。
- 下一阶段默认按“单人日常运营”设计，只有 1 个实际使用者，不把多用户、角色权限和审批流放进近期路线图。
- 继续沿用当前后台访问方式：`ADMIN_USERNAME` / `ADMIN_PASSWORD` + `admin_session`；短期不扩新的登录体系。
- 下一阶段的重点不是继续铺更多页面，而是把已有页面收束成更顺手、更稳定、更能复盘的闭环。

## 2. 规划原则

- 优先单人效率：让 `/admin` 成为真正的日常主工作台，而不是“知道该去哪一页”的导航页。
- 优先质量闭环：先让真实反馈、内容质量门控和提示词实验真正影响生成链路，再考虑更大的能力外延。
- 优先前端信息层级：先减少切页、减少重复筛选、减少读原始 JSON 的次数，再补更多展示。
- 优先可观测且可操作：监控页不只展示状态，还要支持快速定位、失败恢复和复盘。
- 继续保留配置边界：只把少量运行参数做成网页热修改；`.env`、密钥和基础设施配置仍保持只读。

## 3. 明确不纳入

- 多用户、角色权限、审批流、登录体系重构
- 在线编辑整份 `.env`
- 在线展示或修改第三方密钥明文
- 再做一套与 `/admin`、`/admin/phase5`、`/admin/phase6` 重复的新后台页

## 4. P0：下一阶段必须收口的事项

> 建议把 P0 作为下一次正式版本收口范围，优先级高于继续扩新入口。

### 4.1 运营与后端能力

- 落地真实反馈 Provider。以现有 `http` Provider 为接入面，先把“已入草稿任务 -> 自动同步表现 -> 回写实验聚合”跑稳定，而不是继续停留在 mock 和手工补录并行状态。
- 补强内容质量卡口。对高风险稿件增加更明确的事实校验 / 风险门控策略，让“高 AI 痕迹”“高风险题材”“审稿不稳定”的稿件更早进入人工判断，而不是在末端再补救。
- 补齐 Phase 4 流水线的可观测时间线。把写稿、审稿、AI 去痕、复审的 `started / completed / skipped / failed` 串成单任务时间线，并把关键输入输出摘要沉淀为可读日志，而不只是散落在审计记录里。
  - `2026-03-10`：已随 `v1.1.1` 发布到服务器，`workspace.timeline`、`/admin/phase5` 与 `/admin` 摘要层已完成 smoke test。
- 明确 AI 去痕的触发与跳过原因。需要把“未达到 AI 痕迹阈值”“没有 rewrite_targets”“风险过高不允许去痕”“改写后无有效变化”等原因结构化落库并向页面透出，避免只能从结果反推。
  - `2026-03-10`：已随 `v1.1.1` 发布到服务器，`generation.ai_trace_diagnosis`、`/admin/phase5` 与 `/admin` 摘要层已完成 smoke test。
- 推进 Phase 7F 第一刀。先补趋势图、告警分级、去重与静默，重点解决“知道哪里异常”和“避免重复打扰”两个问题。
- 梳理失败恢复主路径。把 Phase 5 的人工动作、推草稿冲突、反馈同步失败、worker 异常的恢复入口做得更统一，减少靠复制错误文本排查。

### 4.2 前端交互与页面逻辑

- 把 `/admin` 收成单人主工作台。首屏默认优先呈现“待人工处理 / 待推草稿 / 今日失败 / 异常堆积”，不再按“多用户分工”设计“我的待处理”之类入口。
- 把“下一步”提示从一句文案升级成结构化动作卡。至少要同时表达：当前卡点、建议动作、要去的页面、为什么这样做。
- 统一后台会话失效恢复。页面检测到 `401` 后，不只提示刷新，还要保留当前 `task_id`、筛选条件和最近输入，让刷新后能回到原上下文。
  - `2026-03-10`：`/admin` 已落地本地持久化与 401 恢复提示，当前会保留 `task_id`、主筛选、搜索词和未提交链接；Phase 5 / Phase 6 / settings / console 仍需继续统一。
- 把参考文章系统做成可点击、可查看的工作区模块。每条入选同题素材至少展示标题、来源、筛选原因和原始链接，并支持一键打开原文或在页面内快速预览。
  - `2026-03-10`：已随 `v1.1.1` 发布到服务器，`workspace.related_articles`、`/admin/phase5` 与 `/admin` 摘要层已完成 smoke test。
- 优化 Phase 5 审核台的信息层级。首屏先给“是否建议通过 / 是否允许推稿 / AI 痕迹等级 / humanize 情况 / 主要风险点”，完整 diff、完整审稿 JSON 和审计轨迹下沉到折叠区。
- 给审核台补“驳回重写后的历史稿对比与回退选择”。当最新稿被驳回或重写后，应能直接对比历史 generation、当前重写稿和最近已通过稿，并允许人工明确选择保留哪一版。
  - `2026-03-10`：已随 `v1.1.1` 发布“采用此版本”与当前采用版本解析，并完成服务器 smoke test。
- 给单任务工作区补完整流水线视图。除了任务状态和审计轨迹，还应能看到写稿摘要、审稿结论、AI 去痕是否触发、触发后改了哪些 block、为什么跳过或失败。
  - `2026-03-10`：已随 `v1.1.1` 发布到服务器，`workspace.timeline`、`workspace.selected_generation`、`/admin/phase5` 与 `/admin` 摘要层已完成 smoke test。
- 优化 `/admin/console` 的模式切换。默认突出日常关注项，只有在排障模式下再展开 SSE、原始快照、队列深度和 worker 心跳等技术细节。

### 4.3 验收标准

- 已入草稿任务可以通过真实反馈 Provider 稳定同步至少一个观察窗口的数据。
- 参考文章列表可以直接点开原文，并能在审核时回看其与当前稿件的关系。
- 监控页可以区分普通提醒和高优先级异常，并支持去重或临时静默。
- `/admin` 首屏无需跳页即可明确看到当前最值得处理的任务集合。
- 会话失效后刷新页面，仍能恢复原来的筛选和当前任务上下文。
- 审核最新一稿时，不需要先展开原始 JSON 才能完成通过 / 驳回 / 推稿判断。
- 驳回重写后，可以直接比较历史版本与当前版本并做人工取舍。
- 页面内可以直接看出 AI 去痕是否触发、为什么没触发、为什么被跳过，以及触发后改动了哪些 block。

## 5. P1：中期整合与效率提升

### 5.1 运营与后端能力

- 给 Phase 6 补 Excel 上传与解析，保留 CSV 作为底层兼容格式，让反馈导入更贴近日常操作。
- 把 Prompt / feedback 数据模型正式化，逐步从 generation 上的字符串字段过渡到更可追踪的版本模型。
- 让 `prompt_experiments` 真正接回生成链路，让实验结果、风格资产和 Phase 4 写稿策略形成闭环，而不是只停留在榜单展示。
- 给 generation diff 增加可复用的摘要层。优先沉淀“标题变化、摘要变化、正文增删规模、主要改写区域”，而不是默认整页直接上全文 unified diff。
- 给参考文章和 generation 建立更稳定的对照关系，让人工审核时可以回看“这段观点主要参考了哪些素材”。

### 5.2 前端交互与页面逻辑

- 打通 `/admin`、`/admin/phase5`、`/admin/phase6` 的上下文跳转。深链应自动带上当前任务、筛选和最近操作，不让人每次重新找任务。
- 给单人场景增加固定视角与最近使用视图，例如“当前待处理”“今天失败”“待推草稿”“最近补过反馈”，并允许记住默认视角。
- 把 Phase 6 重构成“两层工作流”：首层处理缺失反馈与同步任务，二层再看 Prompt 实验榜和风格资产沉淀。
- 统一跨页面状态语言。把系统状态、任务状态、推荐动作、最近操作结果这几层表达方式做成一致模型，减少每页都重新理解一套提示文案。
- 把 Phase 4 的详细时间线做成可折叠的步骤视图，默认看摘要，需要时再展开到日志细节、原始决策和版本差异。

### 5.3 验收标准

- 从 `/admin` 跳到 Phase 5 或 Phase 6 时，任务上下文和筛选条件可自动继承。
- Phase 6 可以在页面内完成“查缺口 -> 同步/导入 -> 看结果 -> 再回任务”的闭环。
- generation diff 默认先展示摘要，再按需展开全文差异。
- 新生成稿能够明确追溯到所用的 prompt 版本、实验状态和资产来源。
- 审核页和工作区可以把参考素材、历史稿件、当前稿件和流水线时间线放在同一上下文里联动查看。

## 6. P2：后续深化项

### 6.1 运营与后端能力

- 补人工重生成 / 深度 humanize 工作区，让高价值稿件可以做更细粒度的人工介入。
- 增加更完整的浏览器级集成回归，覆盖 `/admin`、Phase 5、Phase 6、settings 和 console 的关键操作流。
- 完善配置治理，增加设置变更 diff 历史、影响范围说明和更清晰的回退记录。
- 扩展趋势分析与时间序列指标，支持按时间窗口、来源、状态流转看整体稳定性变化。

### 6.2 前端交互与页面逻辑

- 把现有 `localStorage` 草稿能力升级成显式的“单人工作记忆”，例如恢复上次处理中任务、恢复常用筛选、保留最近反馈窗口。
- 给 console 增加更适合复盘的可视化层，例如异常时间线、队列积压变化和任务流转漏斗。
- 给审计轨迹和运行参数页增加更强的时间维度与变更前后对照，而不是只看当前值。

### 6.3 验收标准

- 关键后台操作流具备浏览器级回归验证，避免页面交互改动造成隐性回退。
- 单人操作者重新进入系统后，能快速恢复上一次工作上下文。
- 监控和设置页可以支撑“发现异常 -> 回看变化 -> 采取动作 -> 留痕复盘”的完整闭环。

## 7. 建议推进顺序

1. 先完成 P0 的真实反馈 Provider、质量卡口和 Phase 7F 基础观测。
2. 同步推进 `/admin`、Phase 5、`/admin/console` 的前端收束，把高频操作尽量留在主工作台和审核台。
3. 再做 P1 的反馈导入升级、Prompt 版本正式化和跨页面上下文打通。
4. 最后进入 P2 的浏览器回归、配置治理和更细的可视化复盘。

## 8. 版本建议

- 如果要按正式版本收口，建议把 P0 视为 `v1.2.0` 的目标范围。
- `v1.2.0` 的标志不应是“又多了几个页面”，而应是：
  - 真实反馈已经接回主链路
  - 告警与趋势观测具备最小可用形态
  - `/admin` 真正承担单人日常主入口
  - 审核和反馈页面的信息层级明显收束
