# v1.1.0 之后的任务规划

更新时间：2026-03-09
状态：Implemented Locally

## 1. 目标

这份文档不是重复路线图，而是把下一阶段里最具体、最紧迫的 4 个问题拆成可执行任务包，作为后续实施前的统一入口。

当前规划只覆盖这 4 个任务：

- 参考文章系统给出具体链接，并支持点击查看
- 驳回重写时可查看历史稿并做人工选择
- 有地方查看文章处理流水线的详细日志或关键信息
- AI 去痕未触发时，能明确看到原因

## 2. 本轮已完成

- `workspace` 现在返回 `related_articles`、`selected_generation`、`timeline`
- generation 现在返回 `ai_trace_diagnosis`、`is_selected`、`draft_saved`、`wechat_media_id`
- `/admin/phase5` 现在可直接点开参考文章、查看 AI 去痕诊断、查看流水线时间线，并支持“采用此版本”
- `/admin` 现在会展示当前采用版本、参考文章、AI 去痕诊断和时间线的摘要层
- `push-wechat-draft`、反馈导入、反馈同步现在都会优先跟随“当前采用版本”
- 已补齐对应测试与阶段文档

以下 5.1 到 5.4 的“当前现状 / 范围 / 验收标准”保留为实施前设计记录；实际落地结果以上述“本轮已完成”和对应阶段文档为准。

## 3. 当前仍未做

- 本轮不调整多用户、权限模型或登录体系
- 本轮不扩展新的后台页，继续收口现有 `/admin`、`/admin/phase5`、`/admin/console`
- 本轮未补服务器部署与 smoke test 记录

## 4. 当前基线

- `docs/post-v1.1.0-roadmap.md` 已经把这 4 个问题纳入 P0 / P1 方向
- `GET /api/v1/tasks/{task_id}/brief` 已经能返回 `related_articles` 详情，说明参考文章的数据底座已经存在
- `GET /api/v1/tasks/{task_id}/workspace` 已经能返回最近 8 个 `generation` 和最近 25 条 `audit log`
- Phase 4 已经写入这些审计动作：
  - `phase4.generation.started`
  - `phase4.generation.completed`
  - `phase4.review.started`
  - `phase4.review.completed`
  - `phase4.humanize.started`
  - `phase4.humanize.skipped`
  - `phase4.humanize.completed`
  - `phase4.humanize.failed`
- AI 去痕的当前触发条件是：
  - `ai_trace_score >= 70`
  - 必须存在 `rewrite_targets`
  - `policy_risk` 不超过 `PHASE4_POLICY_RISK_MAX`
  - `factual_risk` 不超过 `PHASE4_FACTUAL_RISK_MAX`

## 5. 任务包

### 5.1 TP-01：参考文章可查看

#### 目标

让人工审核和工作区不再只看到“同题素材数量”，而是能直接看到参考文章列表、原始链接和必要摘要，并支持点击查看。

#### 当前现状

- `brief` 接口已能返回 `related_articles` 详情
- 当前工作区更常展示的是 `related_article_count`
- 审核决策时还不能在同一上下文中直接回看参考文章原文

#### 范围

- 后端
  - 评估是否扩展 `workspace` 返回 `related_articles`，或在前端补接 `brief` 接口
  - 确认工作区所需字段最小集：`title`、`url`、`source_site`、`summary`、`published_at`、`selected`
- 前端
  - 在 `/admin` 和 `/admin/phase5` 工作区增加“参考文章”模块
  - 每条素材至少支持：
    - 打开原文
    - 查看摘要
    - 标明来源站点与入选状态
  - 如有必要，补快速预览而不是强制跳新页

#### 设计约束

- 不做新的参考文章管理后台
- 不做批量编辑参考素材
- 优先复用现有 `brief` / `workspace` 数据，不重复造接口

#### 验收标准

- 人工审核时可以直接看到被选入的参考文章列表
- 每条参考文章都有可点击链接
- 页面内可以看出这条素材为什么进入当前任务上下文
- 参考文章模块不会打断现有审核主路径

#### 记录重点

- 最终采用的是“扩 `workspace`”还是“补接 `brief`”
- 页面展示了哪些字段，哪些字段明确不展示
- 链接打开方式、失效链接处理方式、空态文案

### 5.2 TP-02：历史稿对比与人工选择

#### 目标

让“驳回重写”不再只是继续生成下一版，而是允许人工在历史稿、当前稿、最近已通过稿之间做明确比较和取舍。

#### 当前现状

- `workspace` 已能返回最近 8 个 `generation`
- Phase 5 现有 diff 主要偏“最新一版 vs 上一版”
- 当前缺少“驳回后如何回看历史稿并恢复/保留某一版”的明确操作流

#### 范围

- 后端
  - 明确历史稿选择是否只是前端选择展示，还是需要新增“选择某版作为当前审阅基线”的动作
  - 如需新增动作，补对应审计日志和冲突规则
- 前端
  - 在 Phase 5 增加“历史稿对比区”
  - 支持至少 3 类版本对照：
    - 最新重写稿
    - 被驳回前的历史稿
    - 最近一次已通过稿
  - 提供人工选择入口，并明确当前保留结果

#### 设计约束

- 不做真正的富文本编辑器
- 不做版本合并器
- 优先做“比较 + 选择”，不做“手工拼稿”

#### 验收标准

- 驳回重写后，人工可以直接比较至少两版以上的历史稿
- 可以明确知道当前审核到底基于哪一版
- 如果支持保留历史稿，相关动作必须留下审计日志
- 历史稿对比不会覆盖掉已有的最新稿审计事实

#### 记录重点

- 参与比较的版本集合是如何选出来的
- “保留某版”的业务语义是什么
- 是否新增内部接口、冲突规则和审计动作

### 5.3 TP-03：流水线详细日志与时间线

#### 目标

把现有分散的审计记录整理成“单任务流水线时间线”，让人能看清写稿、审稿、AI 去痕、复审到底发生了什么，而不是只能看最终状态。

#### 当前现状

- `workspace` 已返回最近 25 条 `audit log`
- Phase 4 已有 generation、review、humanize 相关审计动作
- 这些记录当前更适合作为排障证据，不适合作为日常工作区信息视图

#### 范围

- 后端
  - 归拢需要暴露的关键流水线动作和字段
  - 评估是否需要在 `workspace` 层做时间线摘要，而不只是原始 `audit_logs`
- 前端
  - 在 `/admin` 或 `/admin/phase5` 工作区增加“流水线时间线”
  - 默认展示阶段摘要，需要时再展开原始 payload
  - 覆盖至少这些步骤：
    - 写稿开始 / 完成
    - 审稿开始 / 完成
    - AI 去痕开始 / 跳过 / 完成 / 失败
    - 复审结果

#### 设计约束

- 不把工作区变成原始日志控制台
- 默认优先展示摘要、结论、关键原因，不默认全量展开 JSON
- 如果后续还要接 `/admin/console`，字段命名应尽量统一

#### 验收标准

- 单任务可以按时间顺序看完整流水线
- 不展开原始日志时，也能知道每一步的结果
- 展开后仍能看到原始动作名和关键 payload
- 时间线与最终任务状态、generation 状态、review 结果保持一致

#### 记录重点

- 最终纳入时间线的事件清单
- 每个事件展示哪些字段、哪些字段只保留在原始 payload
- 是否新增“摘要层”字段或转换逻辑

### 5.4 TP-04：AI 去痕触发原因可见化

#### 目标

让页面能明确解释“为什么触发了 AI 去痕”“为什么没触发”“为什么触发后又跳过”，而不是只看到 `humanize_applied=false`。

#### 当前现状

- 触发阈值当前为 `ai_trace_score >= 70`
- 还必须有 `rewrite_targets`
- `policy_risk` 和 `factual_risk` 也不能超阈值
- 即使进入去痕流程，也可能因为 `no_valid_rewrites` 或 `markdown_unchanged` 被 `skipped`
- 这些原因目前主要分散在代码逻辑和审计动作里，页面没有结构化解释

#### 范围

- 后端
  - 明确触发 / 不触发 / 跳过的原因枚举
  - 评估这些原因是透给 `review` 元数据、`workspace` 摘要，还是流水线时间线事件
- 前端
  - 在工作区增加“AI 去痕诊断”或并入“稿件自然度诊断”
  - 至少展示：
    - `ai_trace_score`
    - 是否达到触发阈值
    - 是否存在 `rewrite_targets`
    - 是否因风险门控跳过
    - 是否进入去痕流程
    - 若跳过，跳过原因是什么
    - 若完成，改写了哪些 `block_id`

#### 设计约束

- 不重新设计整套 Phase 4 审稿模型
- 不把内部 prompt 细节直接暴露到后台页
- 原因文案必须和真实代码判断一一对应

#### 验收标准

- 同一任务可以明确区分“未触发”“已触发并完成”“已触发但跳过”“已触发但失败”
- 页面说明与代码判定条件一致
- 如果 AI 去痕没有运行，页面可以直接指出主要原因
- 如果 AI 去痕运行过，页面可以看到改写的 block 和复审结果

#### 记录重点

- 触发阈值与风险上限来自哪里
- 页面文案如何映射代码中的真实条件
- `skipped` 原因是否需要标准化枚举

## 6. 建议实施顺序

### 6.1 里程碑 M1：先补可观测性

- TP-03：流水线详细日志与时间线
- TP-04：AI 去痕触发原因可见化

原因：

- 这两个任务先做，能先回答“系统现在到底做了什么”和“为什么没做某一步”
- 做完后，后续参考文章与历史稿对比的开发和验收也更容易定位问题

### 6.2 里程碑 M2：再补审核工作区

- TP-01：参考文章可查看
- TP-02：历史稿对比与人工选择

原因：

- 这两个任务都直接作用于人工审核体验
- 完成后，Phase 5 工作区会从“只能看最终结果”升级为“可回看素材、可比较版本、可做选择”
